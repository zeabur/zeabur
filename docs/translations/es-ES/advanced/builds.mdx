# Compilaciones

Zeabur utiliza [zbpack](https://github.com/zeabur/zbpack) como herramienta interna para compilar los servicios de los usuarios, de modo que estos puedan desplegar servicios basados en cualquier lenguaje y framework (entorno de trabajo) con un solo clic sin necesidad de entender los detalles complejos.

Por ahora, [zbpack](https://github.com/zeabur/zbpack) es compatible con todos los lenguajes de programación principales para el desarrollo web, y se ha optimizado específicamente para cada framework de desarrollo popular. Al mismo tiempo, se actualiza constantemente para nuevos lenguajes de programación y frameworks:

- Node.js
- Python
- PHP
- Ruby
- Go
- Java
- .NET
- Rust
- Elixir

Si el lenguaje o framework utilizado por tu servicio no está en la lista anterior, o si encuentras que zbpack tiene algún problema durante el proceso de compilación (build), por favor contribuye con un Pull Request en [GitHub](https://github.com/zeabur/zbpack/pulls) para ayudarnos a mejorar la funcionalidad de zbpack.

Cuando utilizas zbpack para compilar un servicio en un proyecto, este identificará automáticamente el lenguaje y el framework utilizados por el servicio basándose en el código, los archivos de configuración y otra información del proyecto, y de acuerdo con esta información, elegirá la forma adecuada de compilar el servicio.

## Generar plan de compilación

Por ejemplo, supongamos que has desarrollado una aplicación web basada en [Next.js](https://nextjs.org/):

```bash
git clone https://github.com/zeabur/nextjs-template
cd nextjs-template
```

Cuando ejecutas el comando `zbpack` en la carpeta de este proyecto, puedes ver el plan de compilación (build plan) personalizado para el mismo:

```bash
zbpack .

╔══════════════════════════════ Build Plan ═════════════════════════════╗
║ provider         │ nodejs                                             ║
║───────────────────────────────────────────────────────────────────────║
║ nodeVersion      │ 18                                                 ║
║───────────────────────────────────────────────────────────────────────║
║ installCmd       │ pnpm install                                       ║
║───────────────────────────────────────────────────────────────────────║
║ buildCmd         │ pnpm run build                                     ║
║───────────────────────────────────────────────────────────────────────║
║ startCmd         │ pnpm start                                         ║
║───────────────────────────────────────────────────────────────────────║
║ packageManager   │ pnpm                                               ║
║───────────────────────────────────────────────────────────────────────║
║ framework        │ next.js                                            ║
╚═══════════════════════════════════════════════════════════════════════╝
```

En la figura, podemos ver que `zbpack` elige utilizar el `nodejs` Provider (proveedor) como proveedor de la solución para la compilación, lo que significa que hará lo siguiente en el plan posterior:

1. Averiguar qué versión de Node.js utiliza el servicio.
2. Averiguar qué gestor de paquetes utiliza el servicio, ya sea `npm`, `yarn` o `pnpm`.
3. Averiguar qué comando utiliza el servicio para instalar las dependencias.
4. Averiguar qué comando utiliza el servicio para compilar.
5. Averiguar qué comando utiliza el servicio para arrancar.
6. Averiguar qué framework de Node.js utiliza el servicio.
