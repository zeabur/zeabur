# 大容量ファイルの取り扱い

Git 経由で Zeabur にプロジェクトをデプロイすると、リポジトリがダウンロードされてビルドコンテキストとして使用されます。リポジトリ内の大容量ファイル（バイナリファイル、データセット、メディアファイル、機械学習モデルなど）は、ビルド時間を大幅に増加させたり、ビルドの失敗を引き起こす可能性があります。

このガイドでは、大容量ファイルが問題を引き起こす理由と、その対処方法を説明します。

## 大容量ファイルが問題を引き起こす理由

### ビルドコンテキストのサイズ制限

GitHub からデプロイする場合、Zeabur はリポジトリを ZIP アーカイブとしてダウンロードします。このアーカイブには **1 GiB のサイズ制限**があります。リポジトリがこの制限を超えると、ビルドは失敗します。この制限以下であっても、大きなリポジトリはダウンロードと展開に時間がかかるため、ビルドが遅くなります。

### Git LFS 非対応

<Callout type="warning">
Zeabur は現在、Git Large File Storage（Git LFS）をサポートしていません。Git LFS で追跡されているファイルはビルド時に**ダウンロードされません**。これにより、ビルド時またはランタイム時にファイルが見つからないエラーが発生します。
</Callout>

リポジトリで Git LFS を使用している場合、ビルドサーバーは実際のファイル内容ではなく、LFS ポインターファイルのみを受け取ります。これらのファイルを外部ストレージソリューションに移行する必要があります。

### コンテナイメージのサイズ

最終的なコンテナイメージに含まれる大容量ファイルは、サービスのデプロイ、スケーリング、再起動に必要な時間を増加させます。イメージをコンパクトに保つことで、コールドスタートの高速化とリソースの効率的な利用が実現できます。

## 推奨される対応方法

### リポジトリから大容量ファイルを削除する

Zeabur はビルドのためにリポジトリ全体をダウンロードするため、`.dockerignore` や類似の仕組みで大容量ファイルを除外することはできません。リポジトリ自体から大容量ファイルを削除する必要があります。

`.gitignore` を使用して大容量ファイルのコミットを防止してください：

```txt filename=".gitignore" copy
# データセットと大容量メディアファイル
*.csv
*.parquet
data/
assets/videos/

# ビルド成果物
dist/
*.log

# 機械学習モデル
*.bin
*.h5
*.onnx
```

<Callout type="info">
すでに大容量ファイルをコミットしており、リポジトリのサイズを縮小するために Git 履歴から削除したい場合は、[git filter-repo](https://github.com/newren/git-filter-repo) や [BFG Repo-Cleaner](https://rtyley.github.io/bfg-repo-cleaner/) などのツールを使用できます。新しいコミットでファイルを削除するだけでは、Git が履歴にファイルを保持するため、リポジトリのサイズは縮小されません。
</Callout>

### 外部オブジェクトストレージを利用する

画像、動画、機械学習モデルなどの大容量の静的アセットは、外部のオブジェクトストレージサービス（AWS S3、Cloudflare R2、またはその他の S3 互換サービスなど）に保存し、アプリケーション内で URL を使って参照してください。

これにより、リポジトリとコンテナイメージをコンパクトに保ちながら、アプリケーションがランタイム時に必要なファイルにアクセスできます。

### ランタイムデータには Volumes を使用する

アプリケーションがランタイム時に大容量ファイルを生成または必要とする場合（ユーザーのアップロードファイル、キャッシュファイル、データベースなど）、コンテナイメージに含めるのではなく、[永続ボリューム](/data-management/volumes)を使用してください。

Volumes に保存されたファイルは、デプロイやサービスの再起動後も保持され、ビルドコンテキストのサイズに影響しません。

### リポジトリをコンパクトに保つ

以下の一般的なベストプラクティスに従って、リポジトリをコンパクトに保ちましょう：

- **バイナリファイルをコミットしない** — コンパイル済みファイル、実行ファイル、アーカイブは CI でビルドするか、ランタイム時にダウンロードしてください。
- **`.gitignore` を使用する** — ビルド成果物、依存パッケージ（`node_modules`、`venv`）、一時ファイルを除外してください。
- **大規模リポジトリを分割する** — リポジトリに複数の独立したサービスが含まれている場合は、それぞれ別のリポジトリに分割し、[監視パス](/deploy/watch-paths)や[ルートディレクトリ](/deploy/root-directory)を使ってデプロイを管理することを検討してください。
