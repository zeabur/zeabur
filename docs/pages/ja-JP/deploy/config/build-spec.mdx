# ビルドスペック

サービスの設定ページを開くと、「**ビルドスペック（Build Spec）**」というフィールドに `Free · 2 vCPU · 4 GB` のような値が表示されています。このページでは、この値が何を指しているのか、そして——より重要な——何を指していないのかを説明します。

<Callout type="info">
**要点**：「ビルドスペック」は、Zeabur がソースコードをコンパイル・パッケージングしてイメージにするために使う、共有 CI/CD ランナーのスペックを表します。ビルド完了後にあなたのサービスが実際に動作するマシンのスペックでは**ありません**。
</Callout>

## 「ビルドスペック」フィールドの意味

サービスをデプロイするたびに、Zeabur は共有のビルドプール上で次の処理を行う CI/CD ジョブを実行します：

1. ソースコードを取得する
2. [zbpack](/deploy/how-deploys-work) でビルドプランを決定する
3. コンテナイメージをビルドする
4. Zeabur のレジストリにイメージをプッシュする

**ビルドスペック**フィールドは、このビルドジョブに割り当てられた CPU とメモリを示します。スペックが高いほどビルドが速くなる——それだけです。

## 有料プランなのに「Free」と表示されるのはなぜ？

CI/CD ランナーは**共有インフラ**で、各グレードのビルドマシンは内部的にプラン名（`Free`、`Dev`、`Pro` …）でラベル付けされています。このラベルは**どのビルドプールにジョブがルーティングされたか**を示すだけで、あなたのアカウントのサブスクリプションプランを反映するものでは**ありません**。

つまり、Pro プランのユーザーでも、ビルドスペック欄に `Free · 2 vCPU · 4 GB` と表示されることがあります。それは単に今回のビルドが Free ティアのビルドプールに割り当てられただけで、**あなたのサービスがレート制限を受けているとか、制限されているといったこととは一切関係ありません**。

## では、私のサービスは実際どこで動いているのか？

あなたのサービスは、まったく別のインフラ上で動いています：

- **共有クラスター** — Zeabur が AWS / GCP / Tencent Cloud 上で管理するノード群。プランに応じてリソースが割り当てられます。
- **専用サーバー** — [BYOS](/byos/aws) で接続した外部サーバー、または Zeabur から直接レンタルしたサーバー。

実行中のサービスが使える CPU・メモリ・帯域は、あなたの**プラン**と**デプロイ先のサーバー**によって決まります——**「ビルドスペック」フィールドとは完全に独立**しています。

## ビルドスペックはサービスのパフォーマンスに影響する？

**しません。** ビルドスペックが影響するのは、**ビルドジョブがどれだけ早く終わるか**だけです。イメージがビルドされ、サービスが起動した時点でビルドマシンは役目を終え、以下には一切影響しません：

- リクエストのレイテンシ
- ランタイム時の CPU / メモリ
- 同時接続数やコネクション数の上限
- 可用性・アップタイム

実行中のサービスが遅い・スロットリングされている気がする、というときに確認すべきはビルドスペック欄ではありません。サービス自体のランタイムメトリクスと、デプロイ先のプラン／サーバーを確認してください。

## ビルドスペックを上げられる？

はい。有料プランはより高スペックのビルドプールにルーティングされます：

| プラン | ビルド CI スペック |
|--------|------------------|
| Free   | 2 vCPU / 4 GB |
| Dev    | 2 vCPU / 4 GB |
| Pro    | 4 vCPU / 8 GB |
| Team   | 4 vCPU / 8 GB |

最新の割り当ては [プラン](/billing/plans) と [zeabur.com/pricing](https://zeabur.com/pricing) を参照してください。
