# 共有クラスター（廃止）

<Callout type="error">
共有クラスターは **2026 年に廃止されました**。すべての新しいプロジェクトは[サーバー](/server)をご利用ください。
</Callout>

## 共有クラスターとは

共有クラスターは、Zeabur の初期のサーバー提供形態でした。このモデルでは、複数ユーザーのサービスが同じ Kubernetes クラスターにデプロイされ、CPU、メモリ、ネットワークリソースを共有していました。ユーザーはハードウェアを管理する必要がなく、実際のリソース使用量に対してのみ課金されていました。

## 共有クラスター vs. サーバー

| 機能 | 共有クラスター（廃止） | サーバー |
| --- | --- | --- |
| リソース分離 | マルチテナント、共有ハードウェア | 完全専用ハードウェア |
| 課金方式 | 従量制（vCPU / メモリ / トラフィック） | 月額固定料金 |
| サーバーロケーション | 固定リージョン（AWS） | Bring Your Own Host (BYOH)、任意のリージョン |
| パフォーマンスの安定性 | ノイジーネイバーの影響あり | 安定かつ予測可能 |
| ユースケース | 小規模プロジェクト、開発/テスト | 本番環境、高信頼性ワークロード |

## 廃止の理由

サーバーの提供が成熟するにつれ、Zeabur はより優れたリソース分離とパフォーマンス保証を提供するため、共有クラスターをサーバーアーキテクチャに統合しました。

## 移行は必須ですか？

**共有クラスター上の既存サービスは引き続き稼働します。** 現時点での「廃止」は*新しい*サービスを追加できなくなったという意味であり、デプロイ済みのサービスが停止されることはなく、既存サービスの統一的な終了期限も発表されていません。

Zeabur がプロジェクト所有者に移行を積極的に依頼するのは、次のような特定の状況のみです：

- プロジェクトのリソース使用量が恒常的に高く、共有クラスターの他のユーザーに影響している
- 不正利用検知（クローラー、マイニングなど）に該当した
- 共有クラスターでは提供できない機能（専用の固定送信元 IP、高帯域、カスタムネットワーク）が必要

連絡を受けていなければ、移行は**任意**です — ただし[専用サーバー](/server)への移行には実際のメリットがあります：プラットフォームレベルのレート制限がない、パフォーマンスが完全に独立している、共有テナントの制約を受けない、などです。

## Clone Project での移行（推奨）

最も簡単な移行方法は、組み込みの **Clone Project** 機能です。プロジェクト全体 — サービス、環境変数、設定 — を選択したリージョンにコピーします：

1. **サーバーをプロビジョニング** — [Zeabur サーバーページ](https://zeabur.com/servers)でサーバーを購入するか、[自分のサーバーを登録](/server/register)してください。
2. 旧プロジェクトで **Settings → General → Clone Project** に移動します。
3. **Target Region** でサーバーのリージョンを選択します。
4. クローンオプションを設定します — クローン中に旧プロジェクトを稼働させ続けるオプションや、サービスグループと並び順の保持が選べます。
5. 確認してクローンの完了を待ち、新プロジェクトのすべてのサービスが正常に起動・動作することを確認します。
6. **DNS / ドメインを切り替え** — カスタムドメインを新しいサービスに再バインドし、DNS レコードを更新します。
7. トラフィックが完全に新プロジェクトで処理されるようになったら、**旧プロジェクトを削除**します。

<Callout type="warning">
クローンには、データ量に応じて数分から数十分のダウンタイムが発生する可能性があります（旧プロジェクトを稼働させ続けるオプションを有効にした場合を除く）。Volume データなどのステートフルストレージはクローン後に必ず検証してください — データベースは先に[バックアップ](/operations/data/backup-restore)を取っておきましょう。
</Callout>

## 手動移行（代替手段）

クローンではなく再構築したい場合：

1. **データをバックアップ** — [バックアップとリストア](/operations/data/backup-restore)機能を使用して、すべてのステートフルサービス（データベース、永続ストレージ）のバックアップを作成してください。
2. **プロジェクトをエクスポート** — [プロジェクトのエクスポート](/deploy/manage/export-project)機能を使用して、プロジェクトの YAML 設定をエクスポートしてください。
3. **サーバーに再デプロイ** — サーバーのリージョンに新しいプロジェクトを作成し、設定をインポートして、サービスを再デプロイしてください。
4. **データを復元** — 手順 1 のバックアップを新しいサービスに復元してください。
5. **DNS を切り替え** — カスタムドメインの DNS レコードを新しいサービスのドメインに向けてください。
6. **確認とクリーンアップ** — 新しい環境が正常に動作していることを確認したら、共有クラスターから古いサービスを削除してください。

## FAQ

**期限までに移行しなければなりませんか？**
既存サービスについて一般的な期限は発表されていません。Zeabur からあなたのプロジェクトについて連絡があった場合にのみ対応が必要です。ただし、2026 年 4 月 1 日以降、共有クラスターでは新しいサービスを作成できません。

**既存のプロジェクトは停止しますか？**
いいえ。既存サービスはこれまでどおり稼働し続けます。廃止により新しいサービスの追加はできなくなりますが、稼働中のサービスが止まることはありません。

**クローンで Volume データも移行されますか？**
データベースの内容と Volume データはクローン後に検証が必要です — 移行前に必ず[バックアップ](/operations/data/backup-restore)を作成し、欠けているものがあれば手動で復元してください。

<Callout type="info">
移行中に問題が発生した場合は、[コミュニティフォーラム](https://zeabur.com/forum)でお問い合わせください。
</Callout>
