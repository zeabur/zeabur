# 构建规格

打开某个服务的设置页面，你会看到一个叫「**构建规格（Build Spec）**」的字段，显示的内容类似 `Free · 2 vCPU · 4 GB`。本文解释这个字段到底代表什么，更重要的是——它**不是**什么。

<Callout type="info">
**一句话结论**：「构建规格」描述的是 Zeabur 用来把你的源码编译、打包成镜像的那台共享 CI/CD 构建机器。它**不是**你服务构建完成后，实际运行时的机器规格。
</Callout>

## 「构建规格」字段代表什么

每次部署服务时，Zeabur 会在共享的构建资源池上跑一个 CI/CD 任务，负责：

1. 拉取你的源代码
2. 通过 [zbpack](/deploy/how-deploys-work) 生成构建计划
3. 构建容器镜像
4. 把镜像推送到 Zeabur 的 registry

**构建规格**字段显示的就是这个构建任务可以使用的 CPU 与内存。规格越高，构建越快——仅此而已。

## 为什么我是付费用户，却还是显示「Free」

CI/CD 构建机器是一套**共享基础设施**，每一档构建机器内部用方案名称做标签（`Free`、`Dev`、`Pro`⋯⋯）。这个标签只是用来说明**这次构建被派发到了哪一档构建资源池**，**并不会**反映你账号的订阅方案。

换句话说：Pro 用户看到的构建规格依然可能是 `Free · 2 vCPU · 4 GB`，那只代表这次构建被分到了 Free 档的构建资源池。这**和你的服务有没有被限流、被限制毫无关系**。

## 那我的服务到底跑在哪里？

你的服务跑在一套完全独立的基础设施上：

- **共享集群** — Zeabur 托管在 AWS / GCP / 腾讯云上的节点，规格按你的方案分配。
- **专属服务器** — 你通过 [BYOS](/byos/aws) 绑定的外部服务器，或直接从 Zeabur 租用的服务器。

你的服务在运行时能使用的 CPU、内存、带宽，是由你的**方案**和**你部署到的服务器**决定的——**和「构建规格」字段完全无关**。

## 构建规格会影响服务性能吗？

**不会。** 构建规格只影响**构建任务跑多快完成**。一旦镜像构建完成、服务启动，构建机器就会被回收，它完全不会影响：

- 请求延迟
- 运行时的 CPU / 内存
- 并发或连接数上限
- 可用性或 uptime

如果你在排查服务运行时的性能问题，不要看构建规格这个字段，去看服务自身的 runtime metrics 以及你部署到的方案 / 服务器。

## 可以提升构建规格吗？

可以。付费方案会被派发到规格更高的构建资源池：

| 方案 | 构建 CI 规格 |
|------|-------------|
| Free | 2 vCPU / 4 GB |
| Dev  | 2 vCPU / 4 GB |
| Pro  | 4 vCPU / 8 GB |
| Team | 4 vCPU / 8 GB |

最新配额请参考 [方案](/billing/plans) 与 [zeabur.com/pricing](https://zeabur.com/pricing)。
