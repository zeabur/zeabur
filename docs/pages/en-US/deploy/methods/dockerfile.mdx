# Deploying with Dockerfile

Normally, you don't need to write a `Dockerfile` by yourself. Zeabur currently provides fast deployment methods for various popular frameworks, but if your project is not using these frameworks or you want to define your own deployment method, you can use a `Dockerfile` to deploy it.

## Creating a Dockerfile

Create a file named `Dockerfile` or `dockerfile` in the root directory of your project and write your deployment method in it. Finally, make sure you have exposed the corresponding `PORT`.

Zeabur will automatically detect if your project has a `Dockerfile`. If so, it will deploy your project using Docker.

<Callout type="info">
  After the deployment starts, you can see the Docker icon above, which means your project has been deployed using Docker.
</Callout>

![docker-deploy](/deploy/dockerfile/docker-deploy.png)

## Deploying without using Dockerfile

If your project has a Dockerfile but you do not want to use this Dockerfile on Zeabur (for example, letting Zeabur decide the best deployment method of this project), you can set `ZBPACK_IGNORE_DOCKERFILE=true` in the environment variables, or add the following to `zbpack.json`:

```json
{
  "ignore_dockerfile": true
}
```

Zeabur will then ignore your Dockerfile and automatically determine the deployment method.

## Environment Variables

If your `Dockerfile` needs to use environment variables, you can click **Environment Variables** on the service page to add them. For more settings, please refer to [Environment Variables](/en-US/environment/variables).

![Environment Variables](/environment/add-variables.png)

If your `Dockerfile` is written in one stage, or if your environment variables only need to be used in the final stage, you don't need to manually write `ENV`. Zeabur will automatically add it for you.

### ARG

However, if your `Dockerfile` is written in multiple stages ([multi-stage](https://docs.docker.com/build/building/multi-stage/)), and you need to set environment variables before building, you can use `ARG` to set them.

Here, we take nodejs and nginx as an example to write a `Dockerfile` for deployment.

```docker {7-8} copy
FROM node:18-alpine AS builder

WORKDIR /app
COPY . .

## The `BUILDTIME_ENV_EXAMPLE` here will be set before building automatically
ARG BUILDTIME_ENV_EXAMPLE
ENV BUILDTIME_ENV_EXAMPLE=${BUILDTIME_ENV_EXAMPLE}

RUN npm install && \
    npm run build

FROM nginx:alpine

COPY nginx.conf /etc/nginx/conf.d/configfile.template
COPY --from=builder /app/dist /usr/share/nginx/html

ENV \
    PORT=8080 \
    HOST=0.0.0.0

EXPOSE 8080

CMD sh -c "envsubst '\$PORT' < /etc/nginx/conf.d/configfile.template > /etc/nginx/conf.d/default.conf && nginx -g 'daemon off;'"
```

## Specifying Dockerfile to Deploy (Monorepo / Multiple Services from One Repo)

When your repo contains multiple Dockerfiles (e.g. a monorepo deploying several services), Zeabur offers three ways to tell each service which Dockerfile to use.

### Option 1: Auto-match by service name (recommended)

The simplest approach: name your Dockerfile `[service-name].Dockerfile` or `Dockerfile.[service-name]`. Zeabur will automatically pick the matching file based on the service name you set in the dashboard. For example, if your service is named `storage`, Zeabur will use `Dockerfile.storage` (or `storage.Dockerfile`).

![specify-service-name](/deploy/dockerfile/specify-service-name.png)

### Option 2: `ZBPACK_DOCKERFILE_NAME` to specify the suffix

If your Dockerfile is named `<suffix>.Dockerfile` or `Dockerfile.<suffix>` but you don't want the service name to dictate the match, set the suffix explicitly:

```
ZBPACK_DOCKERFILE_NAME=storage
```

Or in `zbpack.json`:

```json
{
  "dockerfile": {
    "name": "storage"
  }
}
```

Zeabur will look for `Dockerfile.storage` or `storage.Dockerfile`.

<Callout type="warning">
  This variable accepts **the suffix only**, not a full filename. Setting `ZBPACK_DOCKERFILE_NAME=Dockerfile.storage` makes Zeabur look for a file named `Dockerfile.Dockerfile.storage`, which won't exist — it will then silently fall back to auto-detection and likely build your project with the wrong builder (e.g. Node.js). If you want to pass a full filename or path, use `ZBPACK_DOCKERFILE_PATH` below instead.
</Callout>

### Option 3: `ZBPACK_DOCKERFILE_PATH` for arbitrary paths

If your Dockerfile is not in the repo root, or its filename doesn't follow the `*.Dockerfile` / `Dockerfile.*` convention, use this variable to specify the **full relative path from the repo root**:

```
ZBPACK_DOCKERFILE_PATH=apps/admin/Dockerfile
```

Or in `zbpack.json`:

```json
{
  "dockerfile": {
    "path": "apps/admin/Dockerfile"
  }
}
```

Zeabur will use that exact file to build, with no suffix matching applied. This is the right option for Dockerfiles in subdirectories or with fully custom filenames.

## Docker Compose

Zeabur currently does not support deploying from Docker Compose YAML.

You can convert your Docker Compose file into a Zeabur Template YAML. For details, see [Creating Templates from YAML](/template/template-format). If you need assistance, feel free to contact us at [community](/get-started/faq-support/help).
