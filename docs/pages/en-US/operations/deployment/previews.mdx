# Service Previews

Zeabur supports preview deployments that let you test changes on a separate branch before merging to production. Each branch gets its own isolated environment with a unique preview URL, making it easy to review and share work in progress.

## How Preview Deployments Work

When you connect a Git repository to Zeabur, you can enable preview deployments for branches other than your production branch. Each time you push to a preview-enabled branch, Zeabur automatically:

1. Builds the service from that branch.
2. Deploys it in an isolated environment.
3. Assigns a unique preview URL.

The preview environment is independent from your production environment — it has its own services and does not affect live traffic.

## Enabling Preview Deployments

To set up preview deployments:

1. Open your project in the Zeabur dashboard.
2. Select the service connected to your Git repository.
3. Go to **Settings**.
4. Enable **Preview Deployments** for the branches you want.

Once enabled, every push to those branches triggers a new preview build and deployment.

<Callout>
  Preview deployments are available for Git-deployed services. Services deployed via Docker image or the marketplace do not support automatic branch previews.
</Callout>

## Preview URLs

Each preview deployment receives an automatically generated URL in the format:

```
https://<service>-<branch>-<project>.zeabur.app
```

You can share this URL with teammates, designers, or stakeholders for review. The preview URL remains active as long as the branch exists and the preview environment is running.

## Environment Isolation

Preview environments are fully isolated from production:

- **Separate services** — Preview environments run their own instances. Changes to a preview do not affect your production deployment.
- **Environment variables** — Preview environments inherit the base environment variables from the production environment, but you can override them as needed.
- **Databases and storage** — If your preview environment needs its own database, you can add a separate database service within the preview environment.

## GitHub Integration

For repositories hosted on GitHub, preview deployments integrate with pull requests:

- A deployment status is posted to the pull request with a link to the preview URL.
- Teammates can click the link directly from the PR to review the changes.

For more details on connecting your GitHub repository, see the [GitHub integration](/deploy/github) documentation.

## Best Practices

- **Use preview deployments for pull request reviews.** Let reviewers see a live version of your changes alongside the code diff.
- **Test with production-like data.** Seed your preview database with representative data to catch issues that only appear with real-world inputs.
- **Clean up unused previews.** Delete merged branches and their preview environments to avoid unnecessary resource consumption.
- **Set environment-specific variables.** Use different API keys or endpoints in preview environments to avoid accidentally modifying production data.
