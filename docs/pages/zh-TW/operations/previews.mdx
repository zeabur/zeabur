# 服務預覽

Zeabur 支援預覽部署，讓你在合併到正式環境之前，可以在獨立的分支上測試變更。每個分支都有自己獨立的環境和唯一的預覽 URL，方便審查和分享進行中的工作。

## 預覽部署的運作方式

當你將 Git 儲存庫連接到 Zeabur 時，可以為正式分支以外的分支啟用預覽部署。每次你推送到啟用預覽的分支時，Zeabur 會自動：

1. 從該分支建置服務。
2. 部署到隔離的環境中。
3. 分配一個唯一的預覽 URL。

預覽環境與正式環境是獨立的——它有自己的服務，不會影響正式流量。

## 啟用預覽部署

設定預覽部署的步驟：

1. 在 Zeabur 儀表板中開啟你的專案。
2. 選取連接到 Git 儲存庫的服務。
3. 前往 **Settings**。
4. 為你想要的分支啟用 **Preview Deployments**。

啟用後，每次推送到這些分支都會觸發新的預覽建置和部署。

<Callout>
  預覽部署適用於 Git 部署的服務。透過 Docker 映像檔或 Marketplace 部署的服務不支援自動分支預覽。
</Callout>

## 預覽 URL

每個預覽部署會收到一個自動產生的 URL，格式如下：

```
https://<service>-<branch>-<project>.zeabur.app
```

你可以將此 URL 分享給團隊成員、設計師或相關人員進行審查。只要分支存在且預覽環境正在運行，預覽 URL 就會保持有效。

## 環境隔離

預覽環境與正式環境完全隔離：

- **獨立服務** — 預覽環境運行自己的實例。預覽環境的變更不會影響你的正式部署。
- **環境變數** — 預覽環境會繼承正式環境的基礎環境變數，但你可以依需求覆寫它們。
- **資料庫和儲存** — 如果你的預覽環境需要自己的資料庫，可以在預覽環境中新增獨立的資料庫服務。

## GitHub 整合

對於託管在 GitHub 上的儲存庫，預覽部署會與 Pull Request 整合：

- 部署狀態會發布到 Pull Request 中，並附上預覽 URL 的連結。
- 團隊成員可以直接從 PR 點擊連結來審查變更。

如需了解連接 GitHub 儲存庫的詳細資訊，請參閱 [GitHub 整合](/deploy/github)文件。

## 最佳實踐

- **使用預覽部署進行 Pull Request 審查。** 讓審查者在查看程式碼差異的同時也能看到變更的即時版本。
- **使用接近正式環境的資料測試。** 在預覽資料庫中填入具代表性的資料，以捕捉只有在真實資料輸入時才會出現的問題。
- **清理不再使用的預覽。** 刪除已合併的分支及其預覽環境，避免不必要的資源消耗。
- **設定環境專屬的變數。** 在預覽環境中使用不同的 API 金鑰或端點，避免意外修改正式環境的資料。
