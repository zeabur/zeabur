# 大型檔案處理

當你透過 Git 在 Zeabur 上部署專案時，你的儲存庫會被下載並作為建置上下文使用。儲存庫中的大型檔案（如二進位檔案、資料集、媒體檔案或機器學習模型）會大幅增加建置時間，甚至可能導致建置失敗。

本指南說明大型檔案為何會造成問題，以及如何處理它們。

## 為什麼大型檔案會造成問題

### 建置上下文大小限制

從 GitHub 部署時，Zeabur 會將你的儲存庫下載為 ZIP 壓縮檔。此壓縮檔有 **1 GiB 的大小限制** — 如果你的儲存庫超過此限制，建置將會失敗。即使在此限制以下，較大的儲存庫也會因為下載和解壓縮時間增加而導致建置速度變慢。

### 不支援 Git LFS

<Callout type="warning">
Zeabur 目前不支援 Git Large File Storage（Git LFS）。透過 Git LFS 追蹤的檔案在建置過程中**不會**被下載，這將導致建置時或執行時出現檔案遺失錯誤。
</Callout>

如果你的儲存庫使用了 Git LFS，建置伺服器只會收到 LFS 指標檔案，而不是實際的檔案內容。你需要將這些檔案移至外部儲存方案。

### 容器映像檔大小

包含在最終容器映像檔中的大型檔案會增加部署、擴展和重啟服務所需的時間。保持映像檔精簡可確保更快的冷啟動和更有效的資源使用。

## 建議做法

### 從儲存庫中移除大型檔案

由於 Zeabur 會下載整個儲存庫進行建置，大型檔案無法透過 `.dockerignore` 或類似機制排除。你應該直接從儲存庫中移除大型檔案。

使用 `.gitignore` 防止大型檔案被提交：

```txt filename=".gitignore" copy
# 資料集和大型媒體檔案
*.csv
*.parquet
data/
assets/videos/

# 建置產物
dist/
*.log

# 機器學習模型
*.bin
*.h5
*.onnx
```

<Callout type="info">
如果你已經提交了大型檔案，並且想從 Git 歷史記錄中移除它們以縮小儲存庫大小，可以使用 [git filter-repo](https://github.com/newren/git-filter-repo) 或 [BFG Repo-Cleaner](https://rtyley.github.io/bfg-repo-cleaner/) 等工具。僅在新的提交中刪除檔案並不會縮小儲存庫大小，因為 Git 會在歷史記錄中保留該檔案。
</Callout>

### 使用外部物件儲存存放大型資源

對於大型靜態資源（如圖片、影片或機器學習模型），請將它們儲存在外部物件儲存服務中（例如 AWS S3、Cloudflare R2 或任何 S3 相容的服務），並在應用程式中透過 URL 引用它們。

這樣可以保持儲存庫和容器映像檔的精簡，同時讓你的應用程式在執行時能存取所需的檔案。

### 使用 Volumes 存放執行時資料

如果你的應用程式在執行時會產生或需要大型檔案（如使用者上傳的檔案、快取檔案或資料庫），請使用[持久化儲存空間](/deploy/../data-management/volumes)，而不是將它們包含在容器映像檔中。

儲存在 Volumes 中的檔案會在部署和服務重啟之間持續保留，且不會影響你的建置上下文大小。

### 保持儲存庫精簡

遵循以下一般最佳實務來保持儲存庫的精簡：

- **不要提交二進位檔案** — 編譯後的檔案、可執行檔和壓縮檔應在 CI 中建置或在執行時下載。
- **使用 `.gitignore`** — 排除建置產物、依賴套件（`node_modules`、`venv`）和暫存檔案。
- **拆分大型儲存庫** — 如果你的儲存庫包含多個獨立服務，考慮將它們拆分為獨立的儲存庫，並使用[觸發路徑](/deploy/watch-paths)或[根目錄](/deploy/root-directory)來管理部署。
