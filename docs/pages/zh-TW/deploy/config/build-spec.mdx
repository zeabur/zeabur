# 建置規格

當你在 Zeabur 部署服務時，**建置**和**運行**其實發生在兩台完全不同的機器上。這篇文件說明的是「建置」那一側。

<Callout type="info">
**一句話結論**：你的程式碼是在 Zeabur 的共享 CI/CD 基礎設施上建置的，不是在你服務實際運行的那台機器上。「建置規格」只會影響建置要多久跑完，對你運行中的服務沒有任何影響。
</Callout>

## Zeabur 在哪裡建置你的程式碼

每一次部署都會在 Zeabur 的共享 CI/CD 基礎設施上觸發一個建置任務，這個任務會：

1. 拉取你的原始碼（來自 Git、CLI 上傳，或容器映像來源）
2. 透過 [zbpack](/deploy/how-deploys-work) 決定建置計畫
3. 建置容器映像檔
4. 把映像檔推送到 Zeabur 的 registry

這台建置機器是短暫存在的——映像檔一旦推送出去，建置機器就會被回收，它永遠不會用來跑你的服務。

## 你的服務實際上跑在哪裡

映像檔建好之後，Zeabur 會把它部署到**你選擇的運行環境**：

- **共享叢集** — Zeabur 代管的節點（AWS / GCP / 騰訊雲），規格依你的方案配置
- **專屬伺服器** — 透過 Zeabur 租用，或透過 [BYOS](/byos/aws) 自行綁定的伺服器

服務運行時能使用的 CPU、記憶體、網路、儲存，是由**這套運行環境**決定的——跟建置機器完全無關。

## 各方案的建置規格

付費方案會把建置任務派送到規格更大的建置機器，所以建置會更快完成：

| 方案 | 建置 CI |
|------|---------|
| Free | 2 vCPU / 4 GB |
| Dev  | 2 vCPU / 4 GB |
| Pro  | 4 vCPU / 8 GB |
| Team | 4 vCPU / 8 GB |

最新配額請參考 [方案](/billing/plans) 與 [zeabur.com/pricing](https://zeabur.com/pricing)。

## 建置規格會影響服務效能嗎？

不會。建置規格決定的只是程式碼編譯、打包的速度，僅此而已。它**不會**影響：

- 請求延遲
- 運行時的 CPU / 記憶體
- 並發數或連線數上限
- 可用性或 uptime

如果你運行中的服務變慢或被限流，應該去看服務的 runtime metrics 與部署到的方案 / 伺服器，而不是這個欄位。
