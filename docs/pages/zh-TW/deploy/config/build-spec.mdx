# 建置規格

當你打開某個服務的設定頁面，會看到一個名為「**建置規格（Build Spec）**」的欄位，內容類似 `Free · 2 vCPU · 4 GB`。這篇文件說明這個欄位指的到底是什麼，以及——更重要的——它**不是**什麼。

<Callout type="info">
**一句話結論**：「建置規格」描述的是 Zeabur 用來把你的原始碼編譯、打包成映像檔的那台共享 CI/CD 建置機器。它**不是**你服務在建置完成後，實際運行時的機器規格。
</Callout>

## 「建置規格」欄位代表什麼

每次部署服務時，Zeabur 會在共享的建置資源池上執行一個 CI/CD 任務，負責：

1. 拉取你的原始碼
2. 透過 [zbpack](/deploy/how-deploys-work) 決定建置計畫
3. 建置容器映像檔
4. 把映像檔推送到 Zeabur 的 registry

**建置規格**欄位顯示的就是這個建置任務能使用的 CPU 與記憶體。規格越高，建置越快——就這樣而已。

## 為什麼付費方案用戶看到的還是「Free」

CI/CD 建置機器是一個**共享的基礎設施**，每一個等級的建置機器內部用方案名稱來標示（`Free`、`Dev`、`Pro`⋯⋯）。這個標籤只是用來說明**這次建置被派送到哪個建置資源池**，並**不會**反映你帳號的訂閱方案。

換句話說：Pro 用戶的服務在建置規格欄位依然可能顯示為 `Free · 2 vCPU · 4 GB`，因為這次建置被分配到了 Free 等級的建置資源池。這**跟你的服務有沒有被限流、被限制完全沒有關係**。

## 那我的服務到底是跑在哪？

你的服務運行的機器，是完全另外一套基礎設施：

- **共享叢集** — Zeabur 在 AWS / GCP / 騰訊雲上代管的節點，規格依照你的方案分配。
- **專屬伺服器** — 你透過 [BYOS](/byos/aws) 綁定的外部伺服器，或直接向 Zeabur 租用的伺服器。

你的服務在運行時能使用的 CPU、記憶體、頻寬，是由你的**方案**以及**你部署到的伺服器**決定的——**跟「建置規格」欄位完全無關**。

## 建置規格會不會影響服務效能？

**不會。** 建置規格只會影響**建置任務跑多快完成**。當映像檔建好、服務啟動後，建置機器就釋放掉了，它完全不會影響：

- 請求延遲
- 服務運行時的 CPU / 記憶體
- 並發數或連線數上限
- 可用性或 uptime

如果你在排查服務運行時的效能問題，別看建置規格這個欄位，去看服務本身的 runtime metrics 以及你部署到的方案 / 伺服器。

## 我可以提升建置規格嗎？

可以。付費方案會被派送到規格更高的建置資源池：

| 方案 | 建置 CI 規格 |
|------|-------------|
| Free | 2 vCPU / 4 GB |
| Dev  | 2 vCPU / 4 GB |
| Pro  | 4 vCPU / 8 GB |
| Team | 4 vCPU / 8 GB |

最新配額請參考 [方案](/billing/plans) 與 [zeabur.com/pricing](https://zeabur.com/pricing)。
